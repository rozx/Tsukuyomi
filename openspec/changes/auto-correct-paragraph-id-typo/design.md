## Context

当前翻译批量提交流程在 `validateBatchArgs` 与 `validateParagraphsInRange` 之间采用严格 ID 匹配：`paragraph_id` 只要与任务范围中的真实 ID 不完全一致，就会在范围校验阶段失败。实际运行中模型常出现轻微拼写偏差（如漏/多 1-2 个字符），导致本可判定目标的提交被拒绝，增加重试成本。

本次变更约束：

- 仅对当前任务范围（`chunkBoundaries.allowedParagraphIds`）内 ID 做容错，不扩大权限边界。
- 仅允许“小偏差”自动纠正，且必须可唯一判定；歧义情况仍失败。
- 保持现有返回结构兼容，新增告警字段但不改变成功/失败主语义。

## Goals / Non-Goals

**Goals:**

- 在 `add_translation_batch` 中支持 `paragraph_id` 的轻微拼写纠错（编辑距离 <= 2）。
- 自动纠正时保留可观测性：对模型返回明确 warning，包含原始 ID 与修正 ID。
- 与现有校验链兼容：纠正后继续走重复校验、范围校验、内容校验和批处理流程。
- 维持默认安全策略：超阈值或多候选并列最优时拒绝并提示。

**Non-Goals:**

- 不引入跨 chunk/跨任务范围的模糊检索。
- 不改变 `add_translation_batch` 的批次大小、引号校验、前缀校验等既有规则。
- 不新增 UI 层提示逻辑，本次仅定义工具返回中的 warning。

## Decisions

1. 在参数验证阶段加入“ID 规范化”步骤

- 方案：在 `validateBatchArgs` 后、重复校验前，对 `resolvedIds` 基于允许范围做纠错，输出 `normalizedIds` 与 `correctionWarnings`。
- 原因：越早规范化越能让后续重复检测和范围验证复用同一 ID 视图，避免逻辑分叉。
- 备选：在 `processTranslationBatch` 内才纠错。未采用，因为会绕过前置重复/范围校验，增加状态不一致风险。

2. 匹配算法采用 Levenshtein 编辑距离，阈值固定为 2

- 方案：对每个提交 ID 与允许范围 ID 计算编辑距离；选择唯一最小且 <= 2 的候选作为修正目标。
- 原因：规则稳定、可解释，覆盖“漏字/错字/多字”常见错误。
- 备选：前缀匹配或 Hamming 距离。未采用，前缀匹配误判高，Hamming 不支持插删字符。

3. 歧义冲突采用“拒绝而非猜测”

- 方案：若存在多个候选同为最小距离且 <= 2，则视为不可自动纠正，保留原值让范围校验失败。
- 原因：避免把译文写入错误段落。
- 备选：按字典序或首次命中自动选择。未采用，缺乏业务语义保障。

4. 告警输出并入现有 warning/quality_warnings

- 方案：将纠错告警追加到工具响应中的 warning（批次级）与 quality_warnings（质量告警列表）之一，确保模型可见。
- 原因：兼容现有响应协议，避免新增强制字段导致下游解析改造。
- 备选：新增专用字段 `id_corrections`。可读性更好，但需额外契约升级；本次先保持兼容。

## Risks / Trade-offs

- [风险] 允许纠错可能掩盖模型系统性产出错误 ID → [缓解] 告警中包含原始/修正映射，便于模型与日志追踪。
- [风险] 允许范围很大时编辑距离计算成本上升 → [缓解] 本场景范围通常为 chunk 级，数量受控；必要时后续引入长度剪枝。
- [风险] 歧义拒绝会让少数本可猜中的输入继续失败 → [缓解] 优先保障正确性，要求模型按告警重试。

## Migration Plan

- 仅代码级增量变更，无存储结构迁移。
- 灰度方式：直接随版本发布；若观察到误修正风险，可回滚到禁用纠错分支（移除规范化调用）。

## Open Questions

- 是否在后续版本引入结构化 `id_corrections` 字段，便于模型程序化消费。
- 编辑距离阈值是否需要按 ID 长度动态调整（例如短 ID 更严格）。
