## Context

当前的 `normalizeTranslationSymbols` 函数在处理破折号时，第 480-481 行有以下逻辑：

```typescript
normalized = normalized.replace(/—{3,}/g, '——'); // 3个或更多破折号转为双破折号
normalized = normalized.replace(/–{3,}/g, '——'); // 3个或更多短破折号转为双破折号
```

这会将 3 个或更多的连续破折号（em dash 或 en dash）合并为双破折号。用户反馈这种合并不符合预期，需要保留原始数量。

## Goals / Non-Goals

**Goals:**
- 修改破折号规范化逻辑，保留 2 个或更多的连续破折号
- 仅将单个破折号（—、–）规范化为双破折号（——）
- 更新相关单元测试以验证新行为

**Non-Goals:**
- 不修改其他符号的规范化逻辑
- 不改变函数的签名或返回类型
- 不引入新的配置选项

## Decisions

### 决策 1：移除多个破折号的合并规则

**方案**：删除将 3 个或更多破折号合并为双破折号的正则替换规则。

**理由**：
- 保留原文的排版意图
- 简化逻辑，减少意外的文本转换

**替代方案**：添加配置选项控制此行为（过于复杂，不必要）

### 决策 2：保留单个破折号的规范化

**方案**：继续将单个的 — 或 – 转换为 ——

**理由**：
- 中文排版中破折号应为双字符宽度
- 保持与现有行为的一致性

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 输出行为改变可能影响现有用户 | 这是一个修复错误行为的变更，而非破坏性变更；破折号数量保留更符合用户预期 |
| 可能遗漏其他破折号相关的边界情况 | 通过单元测试覆盖 2 个、3 个、4 个及以上连续破折号的场景 |

## Migration Plan

无需迁移，这是一个纯代码修复，不涉及数据迁移或部署步骤。
