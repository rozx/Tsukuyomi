## Context

当前翻译任务通过 `add_translation_batch` 工具提交批次结果，工具层会做参数与范围校验；任务执行层再从工具调用参数中提取段落并更新内存/进度。现状存在两个跨模块问题：

1. 工具执行与结果消费的“权威来源”不一致：工具可能对参数做修复/校验后成功，但执行层仍可能依赖原始参数做后续提取。
2. 失败反馈主要是自然语言错误，缺少机器可读字段，模型在重试时难以精确补齐缺失 `paragraph_id`，容易进入“修了又错”的循环。

该变更涉及 `translation-tools`、`task-runner`、提示词与测试，属于跨模块行为契约调整，需要先明确统一的结果协议与兼容策略。

## Goals / Non-Goals

**Goals:**

- 定义翻译批次工具的规范化结果协议，使“成功后应用哪些段落”只有一个可信来源。
- 为失败场景提供结构化错误信息，支持模型按字段修复后重试。
- 明确任务执行层的进度推进规则：仅基于成功返回的 accepted items 更新完成状态。
- 保持现有流程可逐步迁移，避免一次性破坏现有调用链。

**Non-Goals:**

- 不改动翻译质量校验逻辑本身（引号、长度、重复译文等规则保持不变）。
- 不引入新的外部依赖或存储结构。
- 不在本次设计中扩展到术语/角色等非翻译批次工具。

## Decisions

### 决策 1：新增规范化成功返回（canonical accepted items）

**选择**

- `add_translation_batch` 成功时返回 `accepted_paragraphs`（数组），每项至少包含：
  - `paragraph_id`
  - `translated_text`

**原因**

- 工具层已经完成合法性与范围校验，最清楚“实际被接受”的结果。
- 任务执行层直接消费该字段可避免再次从原始参数推断，降低重试错配风险。

**备选方案与取舍**

- 备选 A：继续从原始 tool arguments 提取。
  - 缺点：与工具执行结果可能不一致，无法消除修复/重试链路偏差。
- 备选 B：执行层二次复用工具校验函数。
  - 缺点：增加耦合与重复逻辑，仍不如“工具结果即事实”清晰。

### 决策 2：失败返回增加结构化错误字段

**选择**

- 在现有 `success: false` 与 `error` 文本基础上，补充机器可读字段（如 `error_code`、`invalid_items`、`missing_paragraph_ids`）。

**原因**

- 模型可基于字段精确修复（只补缺失/无效项），减少盲目重发整批。
- 便于测试与日志统计，降低对中文错误文案匹配的脆弱性。

**备选方案与取舍**

- 备选：仅优化错误文案。
  - 缺点：仍依赖自然语言理解，不稳定，且难做自动化断言。

### 决策 3：任务执行层优先消费 canonical 结果，保留兼容回退

**选择**

- `task-runner` 在处理 `add_translation_batch` 成功结果时：
  1. 优先读取工具返回的 `accepted_paragraphs`
  2. 若字段缺失（旧格式），回退到原有参数提取路径

**原因**

- 允许分步上线：先升级工具返回，再升级消费端，不要求一次性原子发布。
- 对历史行为保持兼容，降低回归风险。

**备选方案与取舍**

- 备选：强制新格式，无回退。
  - 优点：实现更干净。
  - 缺点：升级窗口期容易出现不可用或行为突变。

### 决策 4：完成度与完整性检查统一基于 accepted items

**选择**

- 仅当工具返回成功时，且仅针对 `accepted_paragraphs` 中的 ID 推进 `submittedParagraphIds` 与段落完成映射。

**原因**

- 避免“提交失败却误计完成”或“部分修复却整批计完成”的状态污染。
- 与 review 阶段缺失校验形成一致语义。

## Risks / Trade-offs

- [风险] 返回 payload 变大（携带 accepted items）→ **缓解**：仅回传必要字段，不附带冗余元数据。
- [风险] 双格式并存导致分支复杂 → **缓解**：限定兼容窗口，后续在稳定后移除回退路径。
- [风险] 错误字段命名不稳定影响提示词与测试 → **缓解**：先固定字段命名与最小集合，再补充可选字段。
- [风险] 旧测试依赖文本错误断言 → **缓解**：保留 `error` 文本，同时新增结构化字段断言。

## Migration Plan

1. 扩展 `add_translation_batch` 成功/失败返回协议（保留现有字段）。
2. 更新 `task-runner`：优先使用 canonical 返回，缺失时回退旧逻辑。
3. 调整提示词，要求模型在失败时依据结构化字段“最小修复后重试”。
4. 增补/更新测试：
   - 工具返回结构测试
   - 执行循环消费 canonical 结果测试
   - 回退兼容测试
5. 观察一轮回归后，再评估移除旧回退路径。

回滚策略：若新协议导致异常，可临时关闭 canonical 消费分支，回退到旧提取路径，同时保留工具层校验与错误提示。

## Open Questions

- `accepted_paragraphs` 是否需要包含附加字段（如规范化后的顺序号）以便调试？
- 结构化错误字段的最小标准集合是什么（必须字段与可选字段边界）？
- 兼容回退路径计划保留多久（按版本还是按指标）？
